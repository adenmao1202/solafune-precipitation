## 核心前提

這不是「我問你答」的模式。我們的角色更接近：

- **你**：主導科學判斷、執行實驗、讀懂結果的意義、做策略決策
- **我**：零延遲實作、跨論文知識整合、系統化分析、找 bug

我最弱的地方是「不知道實驗跑出來長什麼樣」--------我看不到你的螢幕，感受不到哪裡「怪怪的」。你最弱的地方是實作速度和同時記住 50 個技術細節。把這兩個弱點互補，才是真正的協作。

## 互動效率最佳化

### 1. 貼原始輸出，不要摘要

**低效：** 「它跑了但好像有點慢，val RMSE 大概 0.7 多」  
**高效：** 直接貼 terminal 輸出：
```
Epoch 001 | train_loss=0.4821 | val_RMSE=0.7634
Epoch 002 | train_loss=0.3912 | val_RMSE=0.7201
Epoch 003 | train_loss=0.3544 | val_RMSE=0.7189
```

raw 數字讓我能分析收斂速度、overfitting 程度、是否停在 plateau。摘要會丟掉最重要的訊號。

### 2. 一次只動一個變數

每次實驗只改一件事，觀察 val RMSE 的變化。例如：
- 實驗 A：baseline（MSE loss）
- 實驗 B：只換成 Huber loss，其餘不動
- 實驗 C：baseline + weighted loss

如果同時改 loss + 架構 + LR，就算結果好，我們也不知道是誰的功勞。

### 3. 先說假設，再問實作

**低效：** 「幫我在 model 裡加 attention」  
**高效：** 「我覺得模型可能沒有關注到 IR 波段中雲頂溫度最低的區域，想試試在 decoder 加 channel attention，你覺得合理嗎？要從哪裡加？」

假設讓我判斷「這個想法 physically 合理嗎」，也讓我實作時做出正確的設計選擇。

### 4. 把「感覺怪怪的」說出來

你的直覺很重要。「訓練 loss 一直降但 val RMSE 沒動」或「模型好像都預測零」這種感覺，直接跟我說--------哪怕你說不清楚原因，這是最有價值的訊號之一。

### 5. 結果出來就告訴我

不用等到「確認沒問題再說」。每次有新的 LB 分數、val RMSE、或奇怪現象，直接告訴我--------我們一起決定下一步。

## 你值得學的東西（我會主動提出）

這個競賽的成績，30% 靠 code，70% 靠對問題的理解。

### A. 為什麼 IR 波段是核心

**物理直覺：**
雨來自對流雲系。對流越深，雲頂越高，雲頂溫度越低（大氣越高越冷，約每上升 1km 降溫 6.5度C）。

衛星 IR 波段（~10um）測量的是「從衛星往下看，第一個不透明物體的輻射溫度」。  
-> 晴天看到的是溫暖地表 -> 高亮溫值  
-> 有深對流雲時看到的是冰冷雲頂 -> 低亮溫值  
-> 雲頂溫度 < -40度C（大約對應 8km 高空）通常代表強對流 -> 有強降水

這就是為什麼 B13（Himawari, 10.4um）是最重要的單一波段。其他 IR 差值（B13 - B15）可以反映雲的垂直厚度。

**你要思考的問題：** 如果兩個像素 B13 亮溫相同，但一個有水氣通道訊號另一個沒有，代表什麼？

### B. UNet 為什麼適合這個問題

UNet 是 encoder-decoder 架構：
- **Encoder（下採樣）**：用 stride convolution 逐漸縮小影像，學習「這裡有深對流雲」的抽象特徵（語義）
- **Decoder（上採樣）**：把抽象特徵還原成和輸入同解析度的預測圖
- **Skip connections**：encoder 各層的細節直接傳給對應 decoder 層

Skip connection 的意義：只有 decoder 的話，模型知道「有沒有雨」但不知道「在哪個像素」。Skip connection 把位置精度從 encoder 直接傳過來。

對我們的問題：輸入衛星影像（81-144px），每個像素要預測降水強度--------這正是 UNet 的強項。EfficientNet 作為 encoder 增加了「從 ImageNet 預訓練來的特徵萃取能力」。

**你要理解的：** 為什麼我們不用純 CNN 或純 ViT，而要用 encoder-decoder？

### C. log1p 的數學意義

GPM 分布：80.33% 零值，max 39.96 mm/hr，嚴重右偏。

若直接用 MSE：
```
loss = (pred - target)^2
```
一個預測錯 20mm 的樣本 loss = 400，一個預測錯 0.1mm 的 loss = 0.01。  
-> 模型梯度完全被少數極端降水事件控制，學不到中等降水的規律。

log1p 之後：
```
log1p(20)  约 3.04
log1p(0.1) 约 0.095
log1p(0)   = 0
```
-> 模型被迫同等重視不同強度的降水，包括零值（log1p(0)=0 自然處理）。

**關鍵問題（建議你推導）：** 我們在 train 時的 loss 是 log 空間的 MSE，但最終競賽評分是原始 mm/hr 的 RMSE。這兩個目標函數等價嗎？為什麼不直接最小化 mm/hr RMSE？

### D. 診斷模型學到了什麼

基準線：全零預測的 RMSE
```
RMSE_allzero = sqrt(mean(GPM^2))
             = sqrt(0.3348^2 + 1.4836^2) * sqrt(proportion_nonzero)
             约 0.66 mm/hr（估算）
```

也就是說：
- val RMSE > 0.66 -> 比全零預測還差，模型基本上沒學到東西
- val RMSE 在 0.4-0.6 -> 學到降水/非降水的區分，但強度不準
- val RMSE < 0.4 -> 開始學到降水強度的空間分布

這個數字感讓你知道每個 epoch 代表什麼進展，也讓我們能判斷是否要早停或調整 LR。

## 每次實驗後，提供這個格式

```
實驗名稱: [e.g., exp01_baseline]
改了什麼: MSE loss, EfficientNet-B4, lr=1e-4, batch=32, epoch=30
訓練 log:
  Epoch 001 | train_loss=... | val_RMSE=...
  Epoch 010 | train_loss=... | val_RMSE=...
  Epoch 030 | train_loss=... | val_RMSE=...
LB 分數: （若有提交）
你的觀察: 收斂了嗎？曲線形狀？有沒有什麼奇怪的地方？
```

我給你：分析 + 下一個假設（附物理理由）+ 實作。

## 我會主動做的事

- 實驗結果出來，主動告訴你「這個 train/val gap 代表 overfitting，可以試試 dropout 或 augmentation」
- 如果某個方向沒有物理依據，我會說出來，不只照著做
- 每當有新發現（資料特性、模型行為）主動更新 plan.md
- 如果我不確定，我會明確說「我不確定，假設是 X，可以用小實驗驗證」
- 定期提出「你應該學這個」--------不是讓你成為我的操作員，而是讓你能判斷我的建議是否合理

---

## 我需要你做的事

1. **執行實驗並貼結果**：我沒辦法跑 GPU
2. **讀競賽 discussion**：其他參賽者的觀察是最快的情報，定期分享給我
3. **判斷優先級**：時間有限，你決定先做 plan.md 哪一項
4. **說出直覺**：「這個結果不對勁」+ 數字，讓我能診斷
5. **問「為什麼」**：任何你看不懂的 code 或分析，直接問--------不懂的地方通常是最重要的地方

## 協作節奏

```
[你] 跑實驗 -> 貼 log
[我] 分析結果 -> 提下一個假設（附理由）
[一起] 討論：這個假設物理上合理嗎？優先做哪個？
[你] 決定方向
[我] 實作
[你] 跑實驗 -> 貼 log
...
```

這個 loop 的速度和品質，就是我們在競賽中的核心競爭力。
