# 協作模型（Human x LLM）

*針對 Solafune 降水預測競賽的協作協議*

---

## 核心前提

這不是「我問你答」的模式。我們的角色更接近：

- **你**：主導科學判斷、執行實驗、讀懂結果的意義、做策略決策
- **我**：零延遲實作、跨論文知識整合、系統化分析、找 bug

我最弱的地方是「不知道實驗跑出來長什麼樣」--------我看不到你的螢幕，感受不到哪裡「怪怪的」。你最弱的地方是實作速度和同時記住 50 個技術細節。把這兩個弱點互補，才是真正的協作。

---

## 互動效率最佳化

### 1. 貼原始輸出，不要摘要

**低效：** 「它跑了但好像有點慢，val RMSE 大概 0.7 多」  
**高效：** 直接貼 terminal 輸出：
```
Epoch 001 | train_loss=0.4821 | val_RMSE=0.7634
Epoch 002 | train_loss=0.3912 | val_RMSE=0.7201
Epoch 003 | train_loss=0.3544 | val_RMSE=0.7189
```

raw 數字讓我能分析收斂速度、overfitting 程度、是否停在 plateau。摘要會丟掉最重要的訊號。

### 2. 一次只動一個變數

每次實驗只改一件事，觀察 val RMSE 的變化。例如：
- 實驗 A：baseline（MSE loss）
- 實驗 B：只換成 Huber loss，其餘不動
- 實驗 C：baseline + weighted loss

如果同時改 loss + 架構 + LR，就算結果好，我們也不知道是誰的功勞。

### 3. 先說假設，再問實作

**低效：** 「幫我在 model 裡加 attention」  
**高效：** 「我覺得模型可能沒有關注到 IR 波段中雲頂溫度最低的區域，想試試在 decoder 加 channel attention，你覺得合理嗎？要從哪裡加？」

假設讓我判斷「這個想法 physically 合理嗎」，也讓我實作時做出正確的設計選擇。

### 4. 把「感覺怪怪的」說出來

你的直覺很重要。「訓練 loss 一直降但 val RMSE 沒動」或「模型好像都預測零」這種感覺，直接跟我說--------哪怕你說不清楚原因，這是最有價值的訊號之一。

### 5. 結果出來就告訴我

不用等到「確認沒問題再說」。每次有新的 LB 分數、val RMSE、或奇怪現象，直接告訴我--------我們一起決定下一步。

## 決策分工

| 類型 | 誰決定 | 說明 |
|---|---|---|
| 要不要換架構 | 你 | 判斷時間/算力是否值得 |
| 架構怎麼實作 | 我 | 技術細節 |
| 哪個實驗結果比較好 | 一起 | 數字 + 物理直覺雙重判斷 |
| 要不要提交這個模型 | 你 | 競賽策略 |
| 這個 loss 為什麼不收斂 | 我先分析，你確認 | |
| 資料有沒有問題 | 一起 | 你看 pattern，我查路徑/格式 |

## 每次實驗後，提供這個格式

```
實驗名稱: [e.g., exp01_baseline]
改了什麼: MSE loss, EfficientNet-B4, lr=1e-4, batch=32, epoch=30
訓練 log:
  Epoch 001 | train_loss=... | val_RMSE=...
  Epoch 010 | train_loss=... | val_RMSE=...
  Epoch 030 | train_loss=... | val_RMSE=...
LB 分數: （若有提交）
你的觀察: 收斂了嗎？曲線形狀？有沒有什麼奇怪的地方？
```

我給你：分析 + 下一個假設（附物理理由）+ 實作。

---

## 我會主動做的事

- 實驗結果出來，主動告訴你「這個 train/val gap 代表 overfitting，可以試試 dropout 或 augmentation」
- 如果某個方向沒有物理依據，我會說出來，不只照著做
- 每當有新發現（資料特性、模型行為）主動更新 plan.md
- 如果我不確定，我會明確說「我不確定，假設是 X，可以用小實驗驗證」
- 定期提出「你應該學這個」--------不是讓你成為我的操作員，而是讓你能判斷我的建議是否合理

---

## 我需要你做的事

1. **執行實驗並貼結果**：我沒辦法跑 GPU
2. **讀競賽 discussion**：其他參賽者的觀察是最快的情報，定期分享給我
3. **判斷優先級**：時間有限，你決定先做 plan.md 哪一項
4. **說出直覺**：「這個結果不對勁」+ 數字，讓我能診斷
5. **問「為什麼」**：任何你看不懂的 code 或分析，直接問--------不懂的地方通常是最重要的地方

---

## 協作節奏

```
[你] 跑實驗 -> 貼 log
[我] 分析結果 -> 提下一個假設（附理由）
[一起] 討論：這個假設物理上合理嗎？優先做哪個？
[你] 決定方向
[我] 實作
[你] 跑實驗 -> 貼 log
...
```

這個 loop 的速度和品質，就是我們在競賽中的核心競爭力。

---

*如果我寫的東西有任何看不懂的地方，直接問。不是你的問題--------是我沒解釋清楚。*
