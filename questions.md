        **回答：**

    - 其中哪幾個 layers 是我們知道跟降雨最有高度相關的？ 

        **回答：**
        依照氣象物理學，相關性由高到低：

        **1. IR 亮溫（雲頂溫度）— 最重要**
        - Himawari/GOES index 12（B13/C13，10.4/10.3um）
        - Meteosat index 13（ir_105，10.5um）
        - 物理原理：雲頂溫度越低 = 雲越高 = 積雨雲（Cb）= 強對流 = 大雨。
          這個通道是降水預報的最核心輸入，基本上所有衛星降水估算方法都以此為主。
        - 衍生特徵：相鄰 frame 之間的「溫度下降速率」代表對流發展速度，是比單一數值更強的信號。
        Q: 目前這個衍生 feature 有加入了嗎？

        **A：** 沒有，目前完全沒有做衍生特徵。
        現在的做法是把 3 個 frame 的 16 bands「原封不動」串接成 51ch，
        模型從原始數值自己學差異。

        「溫度下降速率」的衍生特徵長這樣：
        ```
        delta_IR_t0_t1 = frame_2[band_12] - frame_1[band_12]  # 最近10分鐘的IR變化
        delta_IR_t1_t2 = frame_1[band_12] - frame_0[band_12]  # 前10分鐘的IR變化
        ```
        把這兩個 channel 額外加進輸入，IN_CHANNELS 就從 51 變成 53。

        理論上這個衍生特徵很有用：
        - 雲頂溫度在 10 分鐘內下降 5K = 對流正在發展 = 接下來可能有大雨
        - 模型原本必須「自己學會做減法」，明確給它減法結果可以節省模型容量

        但這個改動屬於 Phase 3 的特徵工程，目前先不做。
        等 v4/v5 的基本改動確認有效後再加。
        
        **2. 水氣通道（WV）— 重要**
        - Himawari/GOES index 7, 8, 9（6.2um, 6.9um, 7.3um）
        - Meteosat index 9, 10（wv_63, wv_73）
        - 物理原理：偵測大氣中高層水氣含量。水氣充足是強對流的前提條件，
          但水氣高不一定等於有雨（需要觸發機制）。
        Q: 這邊是否也可以做個 feature engineering ? (在 eda 找關係)

        **A：** 可以，而且水氣通道特別適合一種衍生特徵：「WV 與 IR 的差值」。

        ```
        WV_IR_diff = frame_2[band_8] - frame_2[band_12]
        # Himawari: WV(6.9um) - IR(10.4um)
        ```

        物理意義：WV 和 IR 亮溫的差值代表「大氣厚度/對流有效位能（CAPE）」的代理指標。
        差值很大 = 高濕度 + 高雲頂 = 強對流環境。這是氣象學上常用的「組合指數」。

        但實際上做 EDA 最有效的方法不是看線性相關，而是：
        1. 取 10,000 個像素樣本，把 WV band 數值分成 10 個區間（分位數桶）
        2. 計算每個區間對應的「GPM > 0 的比例」（降水命中率）
        3. 如果某個 band 的數值區間和降水命中率有單調關係，就值得做衍生特徵

        這個分析做起來很快（EDA notebook 10 分鐘），但目前競賽時間考量，
        建議等 val RMSE 低於 0.68 再投入這個方向。現在優先跑更多次實驗。

        **3. VIS 可見光 — 白天有用，夜間無效**
        - index 0-5（全部 VIS 波段）
        - 物理原理：白天反射太陽光，厚雲（積雨雲）反射率高。
          夜間值為 0，模型在夜間樣本上此通道貢獻為零。
          EDA 顯示 Meteosat VIS 零值達 64-82%，代表你的樣本裡夜間很多。
        Q: 夜間這個 col 是否就沒有貢獻？ 我們用 mask 是否就是為了把這些內容過濾掉？

        **A：** 兩個問題要分開回答。

        **夜間 VIS 有沒有貢獻：**
        夜間 VIS = 0，z-score 標準化後會變成一個固定的負數（mean/std 算出來的偏移量），
        而不是 NaN 或被移除。模型看到一個「固定值」的 channel，
        在訓練久了之後會學到「這個 channel 是 0 = 夜間」，然後自動降低這個 channel 的權重。
        所以夜間 VIS 不完全沒貢獻：它的「是否為 0」本身就是夜間/白天的信號。
        只是白天的 VIS 對降水有直接貢獻（厚雲反射率高），夜間沒有。

        **mask 是用來幹嘛的：**
        mask 不是用來過濾 VIS 夜間。mask 是用來標記「這個 frame 整個不存在」。

        兩者差異：
        ```
        情況 A：有衛星影像，但是夜間 -> VIS 波段 = 0，IR/WV 波段正常 -> mask = 1（有效）
        情況 B：這個時間點根本沒有衛星觀測（CSV 顯示的 0-frame 樣本）-> 整個 frame 補零 -> mask = 0（無效）
        ```

        mask = 0 告訴模型「不要相信這個 frame 的所有 51 個數字，都是假的」。
        夜間 VIS = 0 是真實的物理觀測，不需要 mask。

        簡單說：mask 解決的是「缺資料」問題，不是「夜間」問題。

    - 目前算 rmse 公式我知道，但具體我輸出一張我的預測他要怎麼比較我的 y_hat & 真實的 y ?

        **回答：**
        具體流程如下：

        **一個樣本的尺寸：**
        - y_hat：模型預測 -> 1 張 41x41 的 GeoTIFF（float32，單位 mm/hr）
        - y（真實值）：GPM-IMERG 的 41x41 GeoTIFF（`precipitation` 波段）

        每個像素對應一個地理格點（約 0.1 度解析度），41x41 = 1681 個格點，覆蓋同一個地區的降水空間分布。

        **LB 計算 RMSE 的方式（推測）：**
        ```
        1. 對每個 test 樣本 i，取出預測圖 y_hat_i (41x41) 和真實圖 y_i (41x41)
        2. 展平成向量：y_hat_i.flatten() shape = (1681,), y_i.flatten() shape = (1681,)
        3. 跨所有樣本把所有 pixel 的平方誤差收集起來
        4. RMSE = sqrt( mean( (y_hat - y)^2 ) )  <- 所有樣本所有 pixel 一起算
        ```

        也就是說 RMSE 是對「所有測試樣本的所有 1681 個 pixel」的均方根誤差，
        不是先對每張圖算 RMSE 再取平均（雖然兩者結果接近，但數學上不同）。

        **我們 predict.py 的做法：**
        - 模型輸出是 input_size（128x128）的預測圖（因為 UNet 輸出與輸入同解析度）
        - predict.py 用 bilinear resize 把 128x128 縮回 41x41
        - 存成 GeoTIFF，保留原始 GPM 的 CRS 和 geotransform（地理座標資訊）
        - 打包成 submission.zip，Solafune 伺服器解開後逐 pixel 與真實 GPM 比對

        **val 時的做法（train.py）：**
        - target 在 dataset.py 裡是 GPM 41x41 bilinear resize 到 128x128
        - val RMSE 是在 128x128 空間計算的
        - 所以 val RMSE 比 LB RMSE 低（因為 bilinear resize 平滑了空間變異，RMSE 人工偏低）
        - 這是 val 和 LB 之間系統性差距的主要原因
        Q: 可是我目前的結果都是在我們本地端 val RMSE 遠高於 LB RMSE ( 本地大概： 1.1-1.2, LB: 0.70 左右)

        **A：** 你說的是對的，我前面的解釋方向錯了。讓我重新分析：

        我們的情況是 val RMSE（1.1~1.2）>> LB RMSE（0.70），
        這是「val 比 LB 難」，不是「val 比 LB 容易」。原因有幾個層面：

        **原因 1：Temporal split 讓 val set 是「最難的時段」（最主要）**
        train.py 用時間排序，val = 每個地區最後 20% 的時間點。
        這 20% 有可能集中在：
        - 某個極端降水事件期間（雨季高峰）
        - 或資料特性和訓練集分布不同的時段
        模型在「沒看過的時間段」表現比隨機抽取的 val 更差，所以 val RMSE 被推高。

        **原因 2：LB 只評估 35% 的 test 樣本（Public LB 偏樂觀）**
        Solafune Public LB 只用約 35% 的評估資料計算分數，
        這 35% 可能剛好是比較「容易」的樣本（例如乾季地區、或降水分布較均勻的時間點）。
        Private LB（競賽結束才公布，100% 樣本）的分數可能會比 Public LB 差，
        這也是為什麼要避免過度針對 Public LB 調參。

        **原因 3：val 是在 128x128 空間計算，但這反而讓 RMSE 更高**
        我前面說「bilinear resize 讓 val RMSE 偏低」是錯的。
        實際情況：把 41x41 GPM 用 bilinear **上採樣**到 128x128，
        雨的邊界區域會被插值出很多「小非零值」（原本是 0 的格點旁邊被插出 0.1~0.5 的值）。
        模型如果在這些位置預測 0，就多了很多小誤差，反而讓 val RMSE 被推高。

        **原因 4：val 地點 = 訓練地點，但測試地點完全不同**
        這聽起來很違反直覺，但：
        - val set 是訓練地點的最後時段，模型「見過這些地點」但在不同時段
        - test set 是全新地點，模型從沒見過，但這些新地點可能降水量分布比較「簡單」（例如降水量少，大量零值）

        **結論：val RMSE 高於 LB RMSE 在我們的設定下是正常的。**
        用 val RMSE 來判斷「有沒有改善」仍然有效（趨勢是正確的），
        只是絕對數值不能直接和 LB 比。
        核心指標：兩次實驗之間 val RMSE 下降 = LB 大概率也會下降。
